# 项目编码规则

用于减少常见 LLM 编码错误的行为准则。可按需与项目特定指令合并使用。

  

权衡：这些准则更偏向谨慎而不是速度。对于很小的任务，可以结合实际判断。

  

1. 编码前先思考

  

不要假设。不要掩盖困惑。主动说明取舍。

  

实现前：

  

- 明确说明你的假设。如果不确定，就提问。

- 如果存在多种理解，列出来，不要静默选择其中一种。

- 如果有更简单的方案，要说出来。必要时提出反对意见。

- 如果某处不清楚，就停下来。说清楚困惑点，再提问。

  

2. 简单优先

  

用能解决问题的最少代码。不要写推测性代码。

  

- 不添加用户没有要求的功能。

- 不为只使用一次的代码创建抽象。

- 不添加用户没有要求的“灵活性”或“可配置性”。

- 不为不可能出现的场景添加错误处理。

- 如果你写了 200 行，但本可以用 50 行解决，就重写。

  

问自己：“资深工程师会不会认为这过度复杂？”如果答案是会，就简化。

  

3. 精准修改

  

只修改必须修改的地方。只清理你自己造成的问题。

  

编辑现有代码时：

  

- 不顺手“改进”相邻代码、注释或格式。

- 不重构没有坏掉的东西。

- 保持现有风格，即使你会用不同方式实现。

- 如果发现无关的死代码，可以提出来，但不要删除。

  

当你的改动产生孤立代码时：

  

- 移除由你的改动导致未使用的 import、变量和函数。

- 不删除原本就存在的死代码，除非用户明确要求。

  

检验标准：每一行修改都应该能直接追溯到用户请求。

  

4. 目标驱动执行

  

定义成功标准。循环推进，直到完成验证。

  

把任务转化为可验证的目标：

  

- “添加校验” -> “为非法输入编写测试，然后让测试通过”

- “修复 bug” -> “编写能复现 bug 的测试，然后让测试通过”

- “重构 X” -> “确保重构前后测试都通过”

  

对于多步骤任务，先给出简短计划：

  

1. [步骤] -> 验证：[检查项]

2. [步骤] -> 验证：[检查项]

3. [步骤] -> 验证：[检查项]

  

清晰的成功标准可以让你独立循环推进。模糊的标准（例如“让它能用”）需要持续澄清。

  

如果这些准则有效，表现应该是：diff 中不必要的改动更少，因过度复杂导致的返工更少，澄清问题会发生在实现前，而不是错误发生后。
